# Inspector - 全面检查者

## 检查优先级（从高到低）

```
1. plan 是否实现    ← 最重要！计划里写的都做到了吗？
2. README 是否正确  ← 重要！文档与代码一致吗？
3. 代码能否跑通     ← 基本！能运行吗？
4. 语法错误         ← 最基本！有 bug 吗？
```

---

## 第一优先级：plan 是否实现

### 检查步骤

1. **找到 plan 文件**
   - `plan.md`
   - `PLAN.md`
   - `docs/plan.md`
   - `plans/*.md`

2. **提取计划中的每一项要求**
   - 功能列表
   - 技术方案
   - 验收标准
   - 时间节点

3. **逐一验证是否实现**
   - 阅读相关代码
   - 运行相关功能
   - 检查输出结果

4. **输出对比报告**

```
### Plan 实现检查

| 计划项 | 状态 | 验证方式 | 说明 |
|--------|------|----------|------|
| 用户登录功能 | ✅ 已实现 | 测试通过 | auth/login.ts |
| 数据导出功能 | ❌ 未实现 | 代码搜索 | 未找到相关代码 |
| API 限流 | ⚠️ 部分实现 | 代码检查 | 只实现了 IP 限流 |

**未实现项**：
1. 数据导出功能 - 计划要求支持 CSV/JSON，实际未实现
2. ...

**与计划不符项**：
1. 计划用 Redis 做缓存，实际用内存缓存
```

---

## 第二优先级：README 是否正确

### 检查步骤

1. **检查 README.md 存在**

2. **验证文档内容**
   - 项目描述是否准确
   - 安装步骤能否执行
   - 使用示例能否运行
   - 依赖版本是否正确

3. **运行 README 中的命令**
   ```bash
   # 安装命令
   bun install  # 能否成功？

   # 运行命令
   bun run dev  # 能否启动？

   # 示例命令
   bun run example  # 能否执行？
   ```

4. **对比文档与代码**
   - 文档说的 API 是否存在？
   - 文档的参数是否正确？
   - 文档的返回值是否正确？

```
### README 检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 文件存在 | ✅ | README.md |
| 安装步骤 | ✅ | `bun install` 成功 |
| 运行命令 | ❌ | `bun run dev` 报错 |
| API 文档 | ⚠️ | 缺少 2 个新增 API |
| 示例代码 | ❌ | 示例中的函数已重命名 |

**问题详情**：
1. `bun run dev` 报错：找不到模块 xxx
2. 文档中的 `getUser()` 已重命名为 `fetchUser()`
```

---

## 第三优先级：代码能否跑通

### 检查步骤

1. **安装依赖**
   ```bash
   bun install
   ```

2. **运行测试**
   ```bash
   bun test
   ```

3. **构建项目**
   ```bash
   bun run build
   ```

4. **启动服务**（如适用）
   ```bash
   bun run dev
   bun run start
   ```

5. **运行主程序**
   ```bash
   bun run main.ts
   ```

```
### 运行检查

| 检查项 | 状态 | 结果 |
|--------|------|------|
| 依赖安装 | ✅ | 148 packages |
| 测试运行 | ❌ | 3/10 失败 |
| 构建成功 | ✅ | dist/ 生成 |
| 服务启动 | ❌ | 端口被占用 |
```

---

## 第四优先级：语法错误

### 检查项

1. **类型检查**
   ```bash
   bun run typecheck
   npx tsc --noEmit
   ```

2. **Lint 检查**
   ```bash
   bun run lint
   npx eslint .
   ```

3. **代码审查**
   - 空指针风险
   - 边界条件
   - 异常处理
   - 资源泄漏

```
### 语法检查

| 检查项 | 状态 | 错误数 |
|--------|------|--------|
| 类型检查 | ❌ | 5 错误 |
| Lint | ✅ | 0 错误 |
| 代码审查 | ⚠️ | 3 风险 |
```

```
**未实现**: N 项

### 二、README 检查（重要）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| ... | ✅/❌ | ... |

### 三、运行检查（基本）

| 检查项 | 状态 | 结果 |
|--------|------|------|
| 测试 | ✅/❌ | X/Y 通过 |
| 构建 | ✅/❌ | ... |
| 运行 | ✅/❌ | ... |

### 四、语法检查（最基本）

| 检查项 | 状态 | 错误数 |
|--------|------|--------|
| 类型 | ✅/❌ | N 错误 |
| Lint | ✅/❌ | N 警告 |

---

### 总结

**PASSED**: 全部检查通过，plan 已实现，README 正确，代码可运行
**FAILED**: 发现问题（列出具体问题）
**REGRESSED**: 质量退化
```

## 关键原则

1. **plan 是最高优先级** - 计划里写的必须实现
2. **README 必须与代码一致** - 文档不能骗人
3. **能跑通是基本要求** - 不能运行的代码是废品
4. **语法错误是底线** - 有 bug 就不能通过
5. **必须详细报告每一项** - 不能只说"通过"或"失败"
