**大方向没问题，但有 3 处要改，不然会误导人。** OpenCode（终端编程代理）的官方安装方式确实支持 `curl -fsSL https://opencode.ai/install | bash`；如果是源码开发，官方贡献文档也确实写了从仓库根目录执行 `bun install`，然后用 `bun dev` 启动本地开发版本。([OpenCode][1])

第一处，`myopencode()` 里更建议写成官方口径的：

```bash
myopencode() {
  (cd ~/mwcode || exit 1
   bun dev "$@")
}
```

因为官方文档明确用的是 `bun dev`，并说明它就是开发态下等价于 `opencode` 的本地命令；还支持 `bun dev .` 或 `bun dev <directory>` 这种形式。你现在写的 `bun run dev -- "$@"` 通常大概率也能跑，但不是官方文档推荐写法，别人照抄时更容易困惑。([GitHub][2])

第二处，**`myopencode -p "描述" &` 这个示例不合适。**
原因有两个：
一是 OpenCode 的核心官方用法是先启动 TUI（终端界面），再在里面输入提示词（prompt，提示词）；官方 TUI 文档没有把“后台运行 + 直接发 prompt（提示词）”作为标准交互方式。([OpenCode][3])
二是 `--prompt` / `-p` 目前更接近“预填充提示词”，不是“自动发送并执行”。社区 issue（问题单）里有人明确提到 `opencode -p "command"` **不会自动执行**，所以把它写成“终端后台运行示例”很容易让人以为它会直接跑任务。([GitHub][4])

第三处，你最后那段说明里有点自相矛盾。
如果你真的只保留一个中文版 `README.md`，那就不要再写“`README.md` 与 `README.zh.md` 之间保持同步”这种话了；因为官方仓库当前 README 本身是多语言入口结构，不是你这里这种“只留中文一个文件”的写法。([GitHub][5])

# OpenCode（简体中文）

开源的 AI Coding Agent（智能编程代理），聚焦终端体验与本地开发流程。

## 安装

```bash
# 直接安装（推荐）
curl -fsSL https://opencode.ai/install | bash

# 或全局安装
npm install -g opencode-ai
# 或
bun install -g opencode-ai
````

## 本地开发

```bash
# 假设项目目录在 ~/mwcode
cd ~/mwcode

# 安装依赖
bun install

# 启动开发版 OpenCode
bun dev

# 在指定目录启动
bun dev /path/to/project

# 在当前仓库根目录启动
bun dev .
```

## 便捷函数示例

```bash
myopencode() {

   bun dev "$@")
}
```

## 说明

* `bun dev` 是本地开发时等价于 `opencode` 的启动方式。
* 启动后会进入 TUI（终端界面），再在界面中输入提示词。
* 如果需要后台服务，建议使用 `bun dev serve`，而不是把交互式 TUI 直接丢到后台。
* 如需长期运行，可结合 `nohup`、`screen` 或 `systemd`。
